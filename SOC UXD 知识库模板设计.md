# SOC UXD 知识库模板设计

版本：V0.1
设计依据：参考 `awesome-design-md` 中 Mintlify、Notion、Linear 等文档型产品的页面组织方式，并结合 SOC UXD 网站当前的信息架构。

## 模板目标

SOC UXD 知识库不是资料堆放处，而是交互设计师做新方案时的工作依据。每篇知识内容都应该帮助使用者快速判断：

- 这条知识适用于什么场景。
- 它约束设计师必须怎么做。
- 它的历史依据来自哪里。
- 它和哪些组件、界面、交付规则有关。
- 交付前应该检查哪些点。

## 页面类型

知识库内容分为五类页面，每类页面使用不同模板。

| 页面类型 | 用途 | 归属工作区 |
| --- | --- | --- |
| 规则页 | 记录稳定设计规则，例如界面框架、层级、提示、多语言 | 规范规则 |
| 流程页 | 记录从策划到交互、从 Figma 到 FGUI 的生产流程 | 设计流程 / 交付落地 |
| 资产页 | 记录历史界面、组件、模式、跳转逻辑 | 资产参考 |
| 模板页 | 记录交互文档、状态矩阵、界面结构等可复用写法 | 设计流程 |
| 复盘页 | 记录项目开发经验、返工原因、历史问题 | 交付落地 |

## 统一页面骨架

每篇知识页都应遵循同一个阅读骨架。

```text
标题
├─ 元信息
│  ├─ 所属工作区
│  ├─ 页面类型
│  ├─ 成熟度
│  ├─ 来源
│  └─ 最近更新
├─ 一句话结论
├─ 适用范围
├─ 核心规则 / 核心流程 / 核心结构
├─ 正确示例
├─ 反例 / 禁止事项
├─ 关联资产
│  ├─ 关联 Figma
│  ├─ 关联组件
│  ├─ 关联历史界面
│  └─ 关联 FGUI / 开发约束
├─ 交付检查项
└─ 待补充 / 待确认
```

## 页面布局建议

参考文档型产品的做法，SOC UXD 知识页建议采用稳定的三段式阅读结构：

```text
左侧：工作区导航
中间：正文内容
右侧：页面内锚点 / 元信息 / 关联入口
```

当前网站可以先保留左侧导航 + 中间正文，右侧目录后续再补。不要为了形式增加复杂度。

## 视觉规范建议

- 页面标题保持 24px 到 28px，不使用营销页大标题。
- 正文使用 13px 到 14px，行高 1.7 到 1.85。
- 卡片半径控制在 6px 到 8px。
- 颜色用于状态辅助，但状态必须有文字标签。
- 表格用于对照关系，不用于大段正文。
- 代码块只用于结构、节点命名、流程文本、模板片段。
- 每页最多保留一个高亮摘要区，避免页面被卡片切碎。

## 知识成熟度标签

| 标签 | 含义 |
| --- | --- |
| 已沉淀 | 已经形成可复用规则，可被 AI 和设计师直接引用 |
| 建设中 | 已有方向或部分资料，但还需要补充示例、边界或来源 |
| 待读取 | 只有主题占位，还没有完成资料读取和结构化 |

## 写作原则

1. 先写结论，再写依据。
2. 每条规则必须说明适用范围。
3. 每个模板必须有检查项。
4. 历史 Figma 只作为依据，不直接等同于规范。
5. 组件库内容必须说明适用和不适用场景。
6. 涉及下游落地时，必须标明程序、GUI、拼接、动效、策划注意事项。
