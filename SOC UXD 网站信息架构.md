# SOC UXD 网站信息架构

版本：V0.2
目标：把现有“通用组件网站”升级为 SOC 项目的 UXD 知识中台，用来承载历史界面、交互规范、组件模式、文档模板、Figma/FGUI 落地规则和 AI 辅助设计能力。

## V0.2 当前采用方案：四工作区

V0.1 按知识类型拆出了较多一级模块，适合盘点内容，但作为网站导航仍然偏散。V0.2 改为任务导向，只保留四个工作区：

```text
SOC UXD
├─ 首页
├─ 设计流程
├─ 规范规则
├─ 资产参考
└─ 交付落地
```

这四个入口分别回答设计师最常见的四个问题：

| 工作区 | 回答的问题 | 收纳内容 |
| --- | --- | --- |
| 设计流程 | 我现在要做一个新交互稿，应该从哪里开始？ | 策划输入契约、交互输出契约、知识库页面模板、交互文档模板、撰写规范、检查清单、AI 交互设计流程 |
| 规范规则 | 这个界面应该遵守哪些底层规则？ | 界面框架、颜色字体、界面层级、信息提示系统、多语言、通用交互设计原则、Figma 节点命名 |
| 资产参考 | 有没有类似历史界面或可复用组件？ | 现存界面结构与跳转逻辑、通用组件清单、组件库使用规则、历史 Figma 模式笔记 |
| 交付落地 | 设计稿怎样才能顺利交给 Figma、FGUI 和开发？ | Figma 读写规范、Figma 到 FGUI 工作流、FGUI 命名与节点落地、项目开发经验约束、AI 自动化入口 |

用户最初提出的十类内容在 V0.2 中这样归位：

| 原始内容 | V0.2 归属 |
| --- | --- |
| 现存全部界面的结构元素及跳转逻辑 | 资产参考 |
| 界面框架规则 | 规范规则 |
| 组件库使用规则 | 资产参考 |
| 颜色字体规范 | 规范规则 |
| 界面层级规范 | 规范规则 |
| 信息提示系统 | 规范规则 |
| 交互文档模板、撰写规范 | 设计流程 |
| 通用交互设计原则 | 规范规则 |
| 多语言规范 | 规范规则 |
| 项目开发经验约束 | 交付落地 |

导航原则：

- 一级导航只放工作任务，不放所有知识分类。
- 组件库不再作为网站主轴，只作为“资产参考”的一个入口。
- 待读取内容可以占位，但必须归入对应工作区，避免扩散成新的一级模块。
- 首页负责解释“怎么找答案”，而不是展示完整知识树。

## 设计定位

SOC UXD 网站不应该只是组件清单，而应该回答交互设计过程中的五类问题：

| 问题 | 对应模块 |
| --- | --- |
| 过去类似界面怎么做？ | 界面资产库 |
| 新界面应该遵守什么规则？ | 规范系统 |
| 可以复用哪些组件和模式？ | 组件与模式库 |
| 交互文档应该写到什么粒度？ | 交互文档与流程 |
| 交付给 FGUI、程序、GUI 时要避开什么坑？ | 开发落地与经验约束 |

网站主轴应从“组件分类”调整为“设计生产链路”：

```text
查历史界面依据
→ 理解设计规范
→ 选用组件 / 模式
→ 输出交互文档
→ 交付 Figma / FGUI / 开发
```

## 一级信息架构

```text
SOC UXD
├─ 首页 / 设计地图
├─ 界面资产库
├─ 规范系统
├─ 组件与模式库
├─ 交互文档与流程
├─ 开发落地与经验约束
└─ AI 辅助设计工作台
```

## 0. 首页 / 设计地图

首页用于建立全局认知，不直接堆文档。

### 需要回答

- 当前知识库覆盖了什么？
- 设计师做一个新交互稿时应该从哪里开始？
- 查历史界面、查组件、查规范、查 Figma/FGUI 规则分别从哪里进入？
- 哪些模块已沉淀，哪些还需要继续读取和整理？

### 页面内容

```text
首页 / 设计地图
├─ 快速入口
│  ├─ 新功能交互设计流程
│  ├─ 查历史界面
│  ├─ 查组件用法
│  ├─ 查框架规则
│  ├─ 查字号字色
│  └─ 查 Figma / FGUI 命名规则
├─ 知识库建设状态
│  ├─ 已沉淀
│  ├─ 建设中
│  └─ 待读取
└─ 常用模板
   ├─ 交互文档模板
   ├─ 界面结构模板
   ├─ 状态矩阵模板
   └─ Figma 交付检查表
```

## 1. 界面资产库

界面资产库是历史经验的证据层，负责记录“过去怎么做”，不直接等同于规范。

### 推荐分类

```text
界面资产库
├─ 战局 / 匹配 / 启动
├─ 社交 / 好友
├─ 组队
├─ 制造系统
├─ 容器系统
├─ 公会
├─ 通用界面
├─ PC / 移动双端界面
└─ 其他系统
```

### 单个界面条目模板

```text
界面名称
├─ 所属系统
├─ Figma 来源
├─ 界面目标
├─ 使用场景
├─ 框架类型
├─ 结构元素
├─ 入口 / 出口 / 跳转逻辑
├─ 状态矩阵
├─ 复用组件
├─ PC / 移动差异
├─ 策划依赖字段
├─ 开发注意事项
└─ 可复用设计结论
```

### 建设状态

当前为“建设中”。已有多个代表性 Figma 文件可作为来源，但还需要继续结构化读取，形成可搜索的界面资产条目。

## 4. 交互文档与流程

该模块连接策划输入和交互输出，负责回答“交互文档写什么，写到哪”。

### 推荐分类

```text
交互文档与流程
├─ 策划输入契约
├─ 交互输出契约
├─ 交互文档模板
├─ 撰写规范
├─ 界面清单写法
├─ 流程图写法
├─ 状态矩阵写法
├─ 边界条件写法
├─ 多端差异写法
├─ 开发标注写法
└─ 评审检查清单
```

### 设计目的

该模块应让交互设计师明确：

- 拿到策划文档后先分析什么。
- 哪些字段缺失时应该反问策划。
- 哪些内容属于交互设计必须补齐。
- 哪些内容应作为 GUI、程序、拼接、动效注意事项交付。

## 5. 开发落地与经验约束

该模块负责回答“怎么交付得下去”，重点不是审美，而是降低 Figma、FGUI、程序、拼接、多语言的返工成本。

### 推荐分类

```text
开发落地与经验约束
├─ Figma 到 FGUI 工作流
├─ Figma 节点命名规范
├─ FGUI 组件映射规则
├─ 设计稿可实现性约束
├─ 程序注意事项
├─ GUI 注意事项
├─ 拼接注意事项
├─ 动效注意事项
├─ 多语言实现约束
└─ 历史问题复盘
```

## 导航改造原则

左侧导航应采用知识体系优先，而不是组件分类优先：

```text
SOC UXD
├─ 首页
├─ 界面资产库
├─ 规范系统
├─ 组件与模式库
├─ 交互文档与流程
├─ 开发落地与经验约束
└─ AI 工作台
```

组件目录应被放入“组件与模式库”之下，作为该模块的子内容继续保留。

## 后续扩展规则

后续每新增一类设计规范，建议都按以下结构沉淀：

```text
主题名称
├─ 规则状态：已沉淀 / 建设中 / 待读取
├─ 适用范围
├─ 规则正文
├─ 例外情况
├─ 历史依据
├─ 关联组件
├─ 关联界面
└─ 检查项
```

这样网站、skill、AI 辅助设计流程可以使用同一套知识结构。
