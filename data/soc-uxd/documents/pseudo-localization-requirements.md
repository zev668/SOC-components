# 伪本地化需求文档

# 1 功能概述

伪本地化功能旨在提供一套游戏内的伪本地化模式，用于在翻译返回之前，QA和UXD对于界面问题进行提前识别和修复，并支持对于模式通过配表的方式进行定制化。

通过输入特定的GM指令，将临时替换当前的`language.json`的语言包，将其中的text字段按照一定的规则替换成为特定语言测试**极限长度与排版**的文本，以提前暴露UI界面可能存在的文本截断、重叠等问题。

# 2 功能入口

**入口：**以GM的形式存在与游戏当中

**触发：**输入特定参数格式的文本 `语言代码;伪本地化模式ID`

**填写示例：**

*   `zhtw;0` \-> 切换为繁体中文，使用模式0

*   `arsa;1`  -> 切换至阿拉伯语，使用模式1

### 3.1.2 添加空格

**功能：**根据表格配置的每种语言是否需要添加空格`\u0020`，以及添加空格的频率。

**作用：**

*   测试是否可以正常安全换行

*   测试多行排版与行高


**规则：**按照频率，在捞取频率数量的字符后添加空格，遇到不满字符数的情况直接跳过

**示例：**假设英文配置了需要添加空格，添加频率是7

原本生成的文本 --> `sjlkdjksajkdlasjdklasjdklajlj`

添加空格后的文本 -->`sjlkdjk sajkdla sjdklas jdklajl j`

### 3.1.4 边际情况

### 3.1.5 泰语特殊规则

**说明：**像泰语这种比较特殊的，需要有单独的生成规则，有关泰文的参考资料见[https://zh.wikipedia.org/wiki/%E6%B3%B0%E6%96%87](https://zh.wikipedia.org/wiki/%E6%B3%B0%E6%96%87)；阿拉伯语本身客户端已经做了适配，所以暂时不在这当中。

泰语的一个视觉单位不一定是一个Unicode 字符，因此在测试 UI 的高度极限，需要按照**“基底 + 元音 + 声调”**的结构进行成组捞取，可见下图示意。虽然泰语可以以基底+元音和声调灵活构成，但是为了进行测试，下述在保留了基底+上元音的情况下，对声调和下元音进行一定规则的捞取。

#### 泰语的垂直四层结构

泰语文字是堆叠式渲染，在进行字符捞取时，必须识别并覆盖以下四个垂直层级

*   **Top Level (最顶层)**：对应声调符号；垂直高度的最高点，用于检测 UI 顶部是否截断。

*   **Above Level (上层)**：对应上元音；位于基底与声调之间，增加上半部分字符密度。

*   **Base Line (基底线)**：对应基底辅音；文字的主体，决定水平宽度。

*   **Below Level (下层)**：对应下元音；垂直高度的最低点，用于检测 UI 底部是否截断。


#### 泰语特殊捞取规则

**1 组团捞取**

在生成泰语测试文本时，不能随机捞取单个字符，必须组团捞取。

设`P = 1 - 常规字符占比`，及常规情况下的**特殊字符占比**

*   **Base Line (基底线)**：100%出现，每次必定捞取一个

*   **Above Level (上层)**：100% 出现，为了保持基本的泰语特征

*   **Top Level (最顶层)**：按P出现，模拟垂直高度极限

*   **Below Level (下层)**：按P出现，模拟垂直深度极限


**2 严格顺序捞取**

Base Line  --> Below  Level-->  Above Level --> Top Level

#### 长度规则

目标文本长度遵循默认规则[《伪本地化需求文档》](https://alidocs.dingtalk.com/i/nodes/kDnRL6jAJM36pxE9tw6yNG3xWyMoPYe1?utm_scene=team_space&iframeQuery=anchorId%3Duu_mmq6mpcpzn26vqv6kkb)；

但是计算泰语字符长度的时候，以一个视觉单位作为一个字符计算，即按照**“基底 + 元音 + 声调”**的结构进行成组。
