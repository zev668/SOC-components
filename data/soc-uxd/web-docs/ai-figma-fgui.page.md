# AI 到 FGUI 工作流

这篇文档说明 SOC 项目里，怎样把 Figma 设计稿通过 AI 流水线转成 FairyGUI 可落地资产。

网页稿只讲流程怎么读、哪些节点必须人工确认、交付物如何流转；完整命令、文件结构和阶段细节仍以 AI 约束源为准。

## 这篇流程解决什么问题

Figma 到 FGUI 的难点不是“生成一个 XML”，而是中间每一步都可能丢语义。

- Figma 节点命名不清，会导致生成结果不可维护。
- 原始节点树太复杂，需要先转成更适合生产的 IR。
- AI 可能误判层级和组件边界，需要人工审树。
- 任务拆分顺序不对，会影响子组件和父组件生成。
- 生成后必须校验引用、语义 ID 和残留问题。

这篇流程的目标，是把“从设计稿到 FGUI”的过程拆成可审查、可回退、可增量修复的阶段。

## 从 Figma 到 FGUI 分几步

![AI 到 FGUI 工作流](assets/soc-uxd/ai-figma-fgui.svg)

| 阶段 | 产物 | 作用 |
|---|---|---|
| create | `figma-raw.json`、`fgui-ir.json` | 从 Figma 读取原始节点，并转成中间表示 |
| checktree | 修正后的 IR | 人工检查层级、组件边界、命名和删减 |
| tasks | `ui-generate-tasks.xml` | 把 IR 拆成可执行的组件生成任务 |
| apply | FGUI 包体和组件 XML | 按任务生成、验证、修复 FairyGUI 组件 |
| archive / mapping | 归档文件、ID 映射 | 保留可追溯关系，支持后续增量修改 |

这套流程的关键是：每个阶段都有明确产物，下一个阶段只消费上一个阶段确认后的结果。

## 哪些环节必须人工确认

AI 可以提速，但不能替代交互和实现判断。

| 人工介入点 | 必须确认什么 |
|---|---|
| 结构审查 | 哪些节点保留、合并、拆组件、改命名、标语义 |
| 任务审查 | 生成顺序是否合理，任务描述是否足够明确 |
| 生成后验收 | 引用是否合法，尺寸和层级是否接近 Figma，交互语义是否还在 |
| 残留问题处理 | 无法自动修复的问题交给对应职能确认 |
| 增量修改 | 判断是局部修复、重跑阶段，还是重新从 Figma 创建 |

如果跳过人工确认，短期看起来更快，后面通常会在组件命名、引用关系和可维护性上返工。

## 交付物怎么流转

每个产物都应该能回答“它从哪里来、给谁用、下一步是什么”。

| 交付物 | 给谁用 |
|---|---|
| Figma 原始节点 | 用于追溯设计来源，不直接给生成器当最终结构 |
| IR JSON | 给结构审查和任务生成使用 |
| 任务 XML | 给生成器逐项执行，也给人工确认任务边界 |
| FGUI 组件文件 | 给拼接、程序和后续验证使用 |
| ID 映射 | 给增量更新、问题定位和回查 Figma 使用 |

交付时不要只交最终文件。中间产物越完整，后续越容易定位问题。
