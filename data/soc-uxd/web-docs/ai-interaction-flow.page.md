# AI 交互设计流程

这篇文档说明 SOC 项目里，AI 应该怎样参与交互设计工作。

它不是一份“让 AI 自动画完所有界面”的说明，也不是替代设计师判断的流程。它的定位更具体：当策划案、历史资料、Figma 资产和项目规范比较分散时，AI 负责把输入整理成可讨论、可评审、可继续落地的交互初稿，并且把假设、风险和待确认项明明白白放出来。

## 这篇流程解决什么问题

在 SOC 项目里，交互设计的难点通常不是“画一个页面”，而是要在画之前先回答清楚：

- 策划到底给了哪些确定信息，哪些只是方向。
- 当前功能应该复用哪些历史模式和组件。
- 移动端与 PC 是否只是适配，还是需要重新组织信息。
- 交互稿要写到什么粒度，才能让 GUI、拼接、程序、动效继续做。
- 哪些问题不能由 AI 或交互设计直接决定，必须回到策划或负责人确认。

AI 交互设计流程的目标，是把这些问题变成一条稳定路径：先整理输入，再查历史依据，再产出交互契约，最后用检查清单做交付前验收。

## 什么时候使用这套流程

只要 AI 参与下面任意一种工作，就应该按这套流程执行：

| 场景 | AI 应该做什么 |
|---|---|
| 策划文档转交互设计 | 先抽取目标、入口、规则、状态、文案、数据和待确认项 |
| 旧功能改版 | 先识别必须保留的旧行为，再写本次新增和调整 |
| PC / 移动双端适配 | 判断是纯适配、局部重组，还是需要单独设计 PC UI |
| Figma 交互稿生成 | 先查组件、规范和历史模式，再生成草稿区内容 |
| 交互稿评审 | 按状态、流程、组件、双端、实现注意和开放问题逐项检查 |
| 知识库补充 | 把新规则拆成人读文档、AI 约束源和参考文档索引 |

简单说：AI 参与的不是“灵感随机生成”，而是“有依据地把散乱输入整理成可执行设计输出”。

这条路径里，AI 每一步都要留下依据。没有依据的地方可以提出建议，但不能伪装成已经确认的规则。

## 每一步要产出什么

| 步骤 | AI 要处理什么 | 应该产出什么 |
|---|---|---|
| 读取任务边界 | 用户目标、策划案、指定页面、Figma 链接、平台范围 | 明确本次任务要做什么、不做什么 |
| 归一策划输入 | 目标、玩家、入口、场景、规则、权限、状态、文案、数据 | `策划输入契约` 和待确认项 |
| 检索历史依据 | 历史 Figma、组件资产、界面清单、通用原则、领域模式 | 可复用结构、组件和规则来源 |
| 产出交互输出 | 信息架构、流程、frame 清单、状态矩阵、双端差异 | `交互输出契约` 和下游注意事项 |
| 生成或修改 Figma | 草稿位置、组件复用、节点安全、截图验收 | 可评审的草稿，不覆盖正式资产 |
| 交付前检查 | 流程完整性、异常状态、PC 行为、组件复用、实现注意 | 可提交、待确认或需要退回补充的判断 |

如果用户只给了一个很粗的需求，AI 仍然可以产出第一版，但必须把推断写成“假设”，把阻塞决策写成“待确认”。这比直接停下来更有用，也比把猜测写成定稿更安全。

## AI 能做什么，不能替代什么

AI 适合做结构化、检索、归纳和第一版草稿，不适合替负责人做最终业务决策。

| AI 可以负责 | AI 不应该越界 |
|---|---|
| 从策划案里提取输入字段、缺失项和风险点 | 替策划决定奖励公式、开放条件、商业化阈值 |
| 查找历史页面、组件和规范，给出复用建议 | 把视觉风格、最终图标、最终美术表现定死 |
| 梳理主流程、异常分支、状态矩阵和双端差异 | 替程序决定接口结构、数据库字段或最终实现方案 |
| 给 Figma 生成草稿、标注假设和下游注意事项 | 覆盖正式组件库、历史交互稿或用户未要求修改的节点 |
| 根据检查清单指出遗漏项 | 把待确认问题当作已经确认的设计结论 |

这条边界很重要。AI 的价值是让设计师更快看到问题全貌，而不是把团队协作里的确认环节绕过去。

## 什么时候必须回到人工确认

遇到下面这些情况，AI 可以提出建议，但必须明确标为“待确认”：

| 情况 | 为什么要人工确认 |
|---|---|
| 奖励、价格、解锁、冷却、排序、人数上限等业务规则缺失 | 这些属于策划或运营决策，不能由交互补规则 |
| 历史行为是否保留不清楚 | 旧功能可能有线上依赖，不能只按新稿推断 |
| PC 行为需要单独设计 | hover、快捷键、滚轮、右键、宽度拉伸会影响实现和体验 |
| 多语言、术语、地区资源存在冲突 | 需要翻译、GUI、程序或负责人共同确认口径 |
| Figma 需要改正式页面或源组件 | 这会影响团队资产，必须得到明确授权 |
| 程序实现成本会改变交互方案 | 需要程序确认可行性和接口条件 |

判断原则很简单：如果某个问题会影响最终体验、商业规则、实现成本或正式资产，就不能靠 AI 默认定稿。

## Figma 工作边界

AI 可以协助读写 Figma，但要保持保守。

| 操作 | 默认规则 |
|---|---|
| 生成新方案 | 放在 `AI草稿`、草稿 frame 或命名清楚的临时区域 |
| 修改已有页面 | 只有用户明确要求同步正式页面时才修改 |
| 使用组件 | 先查已有组件和设计系统，再提出新组件需求 |
| 批量调整 | 写明作用范围，完成后用截图或元数据验收 |
| 引用历史依据 | 在交互说明里标注来源，不把历史案例当最终答案 |

Figma 草稿应当让人一眼看出：这是 AI 辅助产物、依据是什么、哪些地方还没确认、下游需要注意什么。

## 交付前快速清单

- [ ] 是否已把策划输入拆成目标、入口、场景、规则、状态、文案、数据和待确认项。
- [ ] 是否查过相关历史模式、组件资产和规范文档。
- [ ] 是否明确本次设计的范围与非范围。
- [ ] 是否覆盖主路径、取消、关闭、返回、成功、失败、空态、加载、错误、无权限等状态。
- [ ] 是否说明 PC / 移动端差异，例如 hover、快捷键、滚轮、右键、宽度拉伸。
- [ ] 是否拆分程序注意、GUI 注意、拼接注意、动效注意和策划注意。
- [ ] 是否把推断写成假设，把阻塞决策写成待确认。
- [ ] 如果操作 Figma，是否没有覆盖正式组件、历史交互稿或用户未要求修改的节点。
- [ ] 是否用交互检查清单做了最后一轮验收。
