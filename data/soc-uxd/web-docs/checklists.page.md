# 交互检查清单

这份清单用于交互稿交付前的最后一轮自查，也适合在 Figma 评审前快速过一遍。

它不是要把每个项目都变成繁重流程，而是帮助设计师在提交前确认四件事：上游输入是否清楚、交互输出是否完整、Figma 操作是否安全、历史依据是否用对。

## 这篇清单解决什么问题

SOC 的交互稿往往不是只交给一个人看。一个页面交出去之后，GUI、拼接、程序、动效、策划都会继续接力。如果交互稿里只写了“页面应该长这样”，下游很容易在下面这些地方返工：

- 策划输入缺字段，交互默认补了一个规则，后续又被推翻。
- 流程只画了主路径，取消、关闭、返回、失败、空态没有交代。
- PC 与移动端差异没有说明，拼接和程序不知道 hover、快捷键、滚轮、右键是否需要支持。
- 组件复用没有检查，画出了一个看似新、其实已有组件能覆盖的结构。
- Figma 直接改到了源组件或正式页面，导致历史资产被覆盖。

这份清单的目标，是在交付前把这些风险提前暴露出来。能确认的就确认，不能确认的就标成“待确认”，不要让缺失信息伪装成已经定稿。

## 什么时候打开这份清单

只要准备把交互稿交给别人继续工作，就应该打开它。不同阶段关注点不同：

| 阶段 | 主要检查什么 | 输出结果 |
|---|---|---|
| 读完策划案后 | 目标、入口、场景、权限、数据、旧行为是否清楚 | 形成待确认项 |
| 画完主流程后 | 主路径、分支、异常、返回和状态保留是否覆盖 | 补齐流程缺口 |
| Figma 初稿完成后 | frame、组件、状态、双端差异和备注是否可执行 | 交给下游评审 |
| 正式评审前 | 是否误改源文件、是否引用历史依据、是否还有阻塞问题 | 决定能否提交 |

## 四道检查门

![交互交付前的四道检查门](assets/soc-uxd/checklists-gates.svg)

这四道门不是固定顺序的流程表，而是一套交付前的风险过滤器。任何一门没有通过，都不要假装交互稿已经完整；应该把问题写入待确认项，或退回对应环节补齐。

## 第一门：策划输入是否足够清楚

先看输入，再看画面。很多交互稿的问题不是页面画得不够细，而是上游依据没有被拆清楚。

| 要检查的内容 | 通过标准 | 没通过时怎么处理 |
|---|---|---|
| 功能目标 | 能说清楚这个功能解决什么问题、服务哪些玩家 | 写入待策划确认，不用设计假设替代 |
| 入口与返回 | 能说明从哪里进入、如何返回、返回后状态是否保留 | 补入口图或返回说明 |
| 使用场景 | 能区分大厅、局内、匹配、非匹配、全局层、弹层等场景 | 标明适用范围，不混用场景规则 |
| 规则与数据 | 权限、解锁、冷却、时间、排序、筛选、阈值有来源 | 标注配置来源或程序依赖 |
| 新旧行为 | 旧功能保留项与本次新增项分开 | 不让下游误删既有逻辑 |

如果某个字段会影响最终交互决策，例如奖励展示、解锁条件、排序优先级、队伍人数上限，就必须显式标成“待确认”。不要把“经验判断”写成已经确认的规则。

## 第二门：交互输出是否能被下游接住

交互稿不是只给评审看的静态图，它要让下游知道如何继续做。

| 下游角色 | 应该能从交互稿里读到什么 |
|---|---|
| GUI | 页面结构、视觉重点、组件状态、资源需求、宽度适配 |
| 拼接 | frame 清单、组件引用、显隐规则、布局拉伸、跳转连线 |
| 程序 | 数据字段、状态机、权限判断、快捷键、持久化、接口依赖 |
| 动效 | 触发条件、反馈意图、强调对象、加载位置、转场位置 |
| 策划 | 待确认项、配置文案、阈值、奖励展示、规则回填点 |

交互输出至少要覆盖主路径、取消、关闭、返回、成功、失败、无数据、无权限、加载中和错误态。PC 页面还要额外说明 hover、pressed、快捷键、滚轮、右键和宽度拉伸是否需要支持。

## 第三门：Figma 修改是否安全

只要涉及 Figma 写入或改稿，就要先确认操作边界。尤其是 AI 辅助生成或批量调整时，安全检查要放在动手前，而不是出问题之后。

| 要检查的内容 | 通过标准 |
|---|---|
| 源文件与节点 | 已记录文件、页面和节点 ID |
| 修改位置 | 新内容放在 `AI草稿`、草稿 frame，或命名清楚的临时区域 |
| 源组件保护 | 没有覆盖正式组件库、历史交互稿和用户未要求修改的节点 |
| 假设说明 | 生成内容附近能看到规则、依据和待确认项 |
| 结果验收 | 大量写入后看过截图或元数据，确认没有错位、漏图、遮挡 |

如果用户只要求“生成一版方案”，默认应该放在草稿区；只有用户明确要求同步正式页面时，才修改已有交互稿或组件。

## 第四门：历史依据是否用对

历史案例是依据，不是模板复制。检查历史资料时，要看它解决的问题是否和当前功能相同，而不是只看视觉形态像不像。

| 要检查的内容 | 通过标准 |
|---|---|
| 相关模式 | 至少看过一个同领域或同结构的历史模式 |
| 复用边界 | 明确哪些可以复用，哪些因为场景不同不能照搬 |
| 双端差异 | 范围包含 PC 或移动端时，检查过对应平台规则 |
| 组件优先 | 发明新 UI 前，先看过通用组件是否能覆盖 |
| 引用方式 | 在交互说明里标明依据来源，而不是把历史图当最终答案 |

如果历史案例会影响当前决策，例如某类弹窗一直保留返回状态、某类列表一直支持筛选、某类奖励一直用特定提示方式，就应该在交互稿里写出引用依据。

## 检查时怎么判断通过

评审前不要只问“有没有画完”，而要问“下游能不能继续做”。每一项检查都可以落到四种状态：

| 状态 | 含义 | 应该怎么处理 |
|---|---|---|
| 通过 | 信息完整，下游可以执行 | 保留在交互稿中 |
| 待确认 | 需要策划、程序、GUI、拼接或动效确认 | 写入待确认项，不阻塞其他部分推进 |
| 退回补齐 | 缺失会影响主要交互决策 | 先补齐再评审 |
| 不适用 | 当前任务确实不涉及 | 简短标注原因，避免被误认为漏项 |

最重要的是区分“缺失”和“不适用”。例如一个纯移动端功能可以没有 PC hover 规则，但应该明确它不涉及 PC，而不是让下游猜。

## 常见漏项

> 漏项一：只检查主流程，没有检查返回后的状态。

应该补充：取消、关闭、返回、重新打开、跨页面返回后是否保留选中项、分页、筛选和输入内容。

> 漏项二：只检查界面状态，没有检查数据来源。

应该补充：文案、数值、奖励、阈值、排序、筛选和标签是否来自策划配置，程序是否需要提供字段。

> 漏项三：只检查最终画面，没有检查下游备注。

应该补充：程序注意、GUI 注意、拼接注意、动效注意、策划注意。交互稿里没有备注，下游就只能猜。

## 交付前快速清单

- [ ] 功能目标、入口、场景、玩家范围已经写清楚。
- [ ] 权限、解锁、条件、冷却、时间、排序和筛选规则有来源。
- [ ] 旧行为保留项和本次新增 / 调整项已经分开。
- [ ] 主路径、取消、关闭、返回、成功、失败、无数据路径已经覆盖。
- [ ] 默认、选中、hover、pressed、disabled、loading、empty、error、无权限和溢出状态已经覆盖。
- [ ] PC / 移动端差异已经说明。
- [ ] 复用组件已经标明；新增组件有原因。
- [ ] 数据 / API 依赖、GUI 需求、拼接需求、动效意图已经分开标注。
- [ ] Figma 修改位置安全，没有覆盖源组件或正式历史稿。
- [ ] 历史依据和组件复用可能已经检查。
- [ ] 所有阻塞最终决策的问题都写入待确认项。
