# 策划输入 / 交互输出契约

这篇文档用于回答一个最常见的问题：当一个新需求从策划侧流入时，交互设计应该先问什么、能决定什么、最终要交付到什么粒度。

它不是要把策划案重新写一遍，也不是让交互设计替 GUI、程序或动效做最终实现决策。它的作用是把“需求描述”整理成“下游能继续工作的交互方案”。

## 这篇规范解决什么问题

在 SOC 项目里，很多返工不是因为界面画错，而是因为一开始没有把边界说清楚：

- 策划案里只写了“已有功能”，但没有说明哪些行为必须保留。
- 文案、数值、条件、权限、排序规则没有标明是否由策划配置。
- 交互稿只画了主流程，没有说明空态、无权限、冷却中、返回后状态保留。
- 下游拿到稿子后，不知道哪些是程序注意、哪些是 GUI 注意、哪些是拼接或动效注意。

这篇规范的目标，就是让每个新交互稿先完成一次“输入清点”和“输出对齐”。

## 什么时候使用这篇规范

只要任务符合下面任意一条，就应该从这篇开始：

| 场景 | 应该做什么 |
|---|---|
| 新功能从策划案进入交互设计 | 先整理策划输入字段，再画流程和页面 |
| 旧功能改版或铺量旧改 | 先标明必须保留的旧行为，再设计本次变化 |
| 需要 PC / 移动双端适配 | 先判断是纯适配，还是需要单独设计 PC UI |
| 涉及弹窗、列表、奖励、队伍、权限、冷却、状态保留 | 必须补齐状态矩阵和异常分支 |
| 交互稿要交给 GUI、拼接、程序、动效继续落地 | 必须在文档里拆分下游注意事项 |

## 策划输入应该先看什么

读策划案时，不要只摘功能描述。交互设计需要先把输入拆成四类：

| 输入类型 | 需要确认的内容 | 不清楚时怎么处理 |
|---|---|---|
| 目标和玩家 | 这个功能解决什么问题、对哪些玩家开放 | 写入待确认项，不能用设计假设替代 |
| 入口和场景 | 从哪里进入、在哪些平台和状态下使用 | 标明 HUD、大厅、弹窗、列表、聊天等入口 |
| 流程和状态 | 主路径、取消、关闭、返回、成功、失败、冷却、无权限 | 至少覆盖主路径和关键异常分支 |
| 数据和配置 | 文案、数值、数量、阈值、奖励、排序、权限 | 明确哪些来自策划配置，哪些需要程序支持 |

如果某个决策明显属于策划边界，例如奖励公式、解锁条件、排序优先级，不要在交互稿里替策划定死。正确做法是把它标成“待策划确认”。

## 交互输出应该交付什么

一份合格的交互输出，至少要让五类下游角色继续工作：

| 下游角色 | 需要从交互稿里读到什么 |
|---|---|
| GUI | 页面结构、视觉重点、组件状态、资源需求、响应式宽度 |
| 拼接 | frame 清单、组件引用、显隐规则、布局拉伸、跳转连线 |
| 程序 | 状态机、数据字段、权限判断、快捷键、持久化、接口依赖 |
| 动效 | 触发条件、反馈意图、强调对象、加载和转场位置 |
| 策划 | 待确认项、配置文案、阈值、奖励展示、规则回填点 |

因此，交互文档里不能只有“页面长什么样”。它还必须说明页面在什么状态下出现、用户操作后发生什么、哪些内容来自配置、哪些行为需要下游实现。

## 做到哪就该停下来

交互设计负责把体验边界定义清楚，但不负责替其他职能完成最终细节。

| 应该由交互决定 | 不应该由交互越界决定 |
|---|---|
| 流程、状态、组件用法、异常分支、跨平台行为 | 奖励公式、业务阈值、最终美术风格、程序实现方案 |
| 返回路径是否保留状态 | 具体接口结构和数据库字段 |
| 是否需要 toast、弹窗、侧边提示、持续提示 | 最终动效曲线、粒子细节、特效资源 |
| PC 是否需要 hover、快捷键、滚轮、右键 | GUI 最终图标绘制和艺术字表现 |

一句话：交互要说清楚“用户应该怎么走、系统应该怎么反馈、下游需要注意什么”；但不要替策划定规则、替 GUI 定画风、替程序定实现。

## 常见误区

> 误区一：策划案里没写，就默认按经验补上。

应该改成：把缺失项列出来，并标注“待策划确认”。尤其是奖励、权限、排序、数值阈值。

> 误区二：只画主流程，异常状态交给下游自己理解。

应该改成：至少补齐无权限、无数据、冷却中、加载中、错误态、返回后状态保留。

> 误区三：旧功能改版时，只画新页面，不说明旧行为哪些保留。

应该改成：明确“保留旧行为”和“本次新增 / 调整行为”，避免下游误删已有逻辑。

## 交付前检查

- [ ] 是否写清楚功能目标、适用玩家、入口和使用场景。
- [ ] 是否覆盖主路径、取消 / 关闭 / 返回路径。
- [ ] 是否覆盖空态、加载态、错误态、无权限、冷却中等关键状态。
- [ ] 是否标明文案、数值、阈值、奖励和条件是否由策划配置。
- [ ] 是否说明 PC / 移动端差异，例如 hover、快捷键、滚轮、右键。
- [ ] 是否拆分程序注意、GUI 注意、拼接注意、动效注意、策划注意。
- [ ] 是否列出会阻塞最终交互决策的待确认项。
