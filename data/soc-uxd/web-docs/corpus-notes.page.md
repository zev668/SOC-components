# 历史经验语料笔记

这篇笔记把 SOC 过往典型 Figma 文档里的交互经验整理成模式索引。它不是正式规范，也不是组件清单，而是“做新设计前先翻一眼”的历史经验入口。

当你面对一个新系统、不确定有没有类似案例时，可以先看这里：哪些功能域已经出现过，哪些页面结构反复使用，哪些双端适配问题容易被遗漏。

## 这篇笔记解决什么问题

新需求最容易从零开始画，但 SOC 已经有大量同类经验。语料笔记要解决的是“历史经验不可检索”的问题。

- 模式入口、战斗入口、组队、好友、制造、背包、公会、赛季等功能域已有相似结构。
- 组件库和通用界面里有可复用的按钮、弹窗、列表、Tips 和选择流程。
- 双端框架里已有移动端转 PC 的常见处理方式。
- 历史稿里出现过的问题，可以转成新交互稿的检查点。

它的价值在于帮设计师快速判断：这次是已有模式复用、已有模式变体，还是确实需要新增规则。

## 目前沉淀了哪些模式

当前笔记更偏模式归纳，不适合直接当交互稿使用。

| 模式 | 使用方式 |
|---|---|
| 入口型页面 | 先判断入口是否承载状态、奖励、倒计时、准入条件 |
| 列表型页面 | 先确认筛选、排序、空态、加载、错误和返回定位 |
| 选择型流程 | 先确认单选 / 多选、已拥有、不可用、数量不足和确认反馈 |
| 制造 / 队列 | 先确认材料消耗、时间、加速、取消、完成和领取 |
| 社交 / 组队 | 先确认身份、权限、邀请、拒绝、超时和重复操作 |
| 双端适配 | 先确认移动端结构是否要转为 PC 顶栏、侧栏、hover 和键位 |

新方案如果落在这些模式内，应优先查对应历史案例和组件规范。

## 用它做新设计时怎么查

使用语料笔记时，不要只复制历史页面。建议按这四步走。

| 步骤 | 目的 |
|---|---|
| 先找功能域 | 判断需求属于战斗入口、制造、背包、社交、组队、公会还是通用框架 |
| 再找页面模式 | 判断是入口、列表、详情、选择、确认、弹窗、HUD 还是结果反馈 |
| 再查组件规范 | 看是否已有通用组件能承载，不要只按截图相似度判断 |
| 最后写待确认 | 把历史稿没有覆盖的新状态、新平台差异和新业务规则标出来 |

如果历史案例和当前需求冲突，以当前策划输入和正式规范为准；语料笔记只提供参考路径。
