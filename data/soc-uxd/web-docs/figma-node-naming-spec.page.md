# Figma 节点命名规范

这篇文档说明 SOC 项目里，进入 FGUI / IR 转换链路的 Figma 节点应该怎么命名。

它不是为了让图层列表看起来整齐，而是为了让后续自动转换、AI 检查、组件复用和人工排查都能稳定识别节点含义。命名写对了，Figma 才能从“设计稿”变成“可解析的交付结构”。

## 这篇规范解决什么问题

Figma 节点命名不稳定，会直接影响下游：

- 顶层 Frame 没有包体名，导出时不知道属于哪个 FGUI 包。
- 按钮、列表、文本、图片都靠猜，转换结果不稳定。
- 已有 FGUI 资源没有写引用，导致重复新建组件。
- 效果图、标注图、参考图没有忽略，被误解析进 FGUI。
- `topbar`、`navbar` 这类大小写不统一，破坏全屏框架结构。

这篇规范的目标，是让每个关键节点都能回答三个问题：它是什么类型、它叫什么、它是否引用已有资源。

## 什么时候必须按命名规范

只要 Figma 内容会进入 FGUI、IR、AI 检查或自动化生成链路，就必须按本规范命名。

| 场景 | 为什么要规范命名 |
|---|---|
| 正式交互稿交付 | 下游需要从节点名识别界面结构和控件类型 |
| Figma 转 FGUI | 转换器需要根据前缀和资源引用生成节点 |
| 复用已有组件 | `@包名/组件名` 可以避免重复造组件 |
| AI 检查设计稿 | AI 需要通过命名判断节点角色 |
| 全屏界面结构 | `bg / root / topBar / navBar / content` 必须保持稳定 |

如果只是临时草图，不进入下游链路，可以先不完整命名；一旦要交付，就要补齐。

## 标准命名公式

![Figma 节点命名公式](assets/soc-uxd/figma-node-naming-spec.svg)

进入转换链路的节点，推荐使用这个公式：

```text
类型前缀#节点名称@资源引用
```

三个片段的含义：

| 片段 | 是否必填 | 作用 |
|---|---|---|
| `类型前缀#` | 可选但推荐 | 告诉转换器映射到哪种 FGUI 类型 |
| `节点名称` | 必填 | 节点在 IR / FGUI 中的语义名称 |
| `@资源引用` | 可选 | 引用已有 FGUI 包体、组件或资源 |

示例：

```text
button#btnBuy@CommonGlobal/BtnBuy
```

含义是：这是一个按钮节点，节点名叫 `btnBuy`，复用 `CommonGlobal` 包里的 `BtnBuy` 资源。

## 顶层 Frame 和全屏结构怎么命名

一个可导出的界面，顶层 Frame 至少要携带包体名：

```text
UiLogin@LobbyLogin
```

其中 `UiLogin` 是界面 / 组件名称，`LobbyLogin` 是 FGUI 包体名。

全屏界面建议使用稳定骨架：

```text
UiLogin@LobbyLogin
  bg
  root
    topBar
    navBar
    content
```

注意三点：

- `topBar`、`navBar`、`content` 是 `root` 下的框架模块。
- `navBar` 是可选结构，不需要导航时不要硬建。
- `topBar` 和 `navBar` 使用小驼峰，旧写法 `topbar`、`navbar` 要修正。

## 类型前缀怎么选

类型明确时，优先显式写前缀。

| 前缀 | 映射类型 | 常见用途 |
|---|---|---|
| 缺省 | `GComponent` | 普通组件或容器 |
| `ignore#` | 不解析 | 效果图、说明图、标注图、参考图 |
| `graph#` | `GGraph` | 几何图形 |
| `img#` | `GImage` | 图片 |
| `loader#` | `GLoader` | 动态装载资源 |
| `text#` | `GTextField` | 普通文本 |
| `rich#` | `GRichTextField` | 富文本 |
| `list#` | `GList` | 列表 |
| `group#` | `Group` | FGUI 分组 |
| `button#` | `GButton` | 按钮 |
| `progress#` | `GProgressBar` | 进度条 |
| `slider#` | `GSlider` | 滑动条 |
| `scrollbar#` | `GScrollBar` | 滚动条 |
| `combobox#` | `GComboBox` | 下拉框 |

不要把资源引用强塞给不支持引用的类型。比如纯文本、几何图形、分组节点，通常不应该写 `@资源引用`。

## 资源引用怎么写

资源引用写在节点名称之后：

```text
节点名称@包名/组件名
```

如果省略组件名：

```text
节点名称@包名
```

则表示使用该包中与节点同名的组件或资源，具体是否允许由转换流程和组件库匹配结果决定。

复用已有资源时，包名和组件名要尽量与 FGUI 真实资源一致。拿不准资源名时，先查组件库，不要临时创造一个看起来差不多的名字。

## 哪些节点要显式忽略

这些节点通常不应该进入 FGUI 转换链路：

- 效果图。
- 交互标注说明。
- 参考图。
- 截图底图。
- 纯阅读说明。
- 设计师临时摆放的对照素材。

这类节点建议使用：

```text
ignore#说明文字
ignore#效果图
```

显式忽略比事后清理更安全。转换器和 AI 不应该靠猜测判断某个图层是不是“只是参考”。

## 交付前快速清单

- [ ] 顶层 Frame 至少带有包体名，例如 `UiExample@PackageName`。
- [ ] 全屏界面使用 `bg / root / topBar / navBar / content` 的标准大小写。
- [ ] 不需要导航的页面没有误建 `navBar`。
- [ ] 关键节点已写清类型前缀。
- [ ] 效果图、标注图、参考图已使用 `ignore#`。
- [ ] 复用已有资源的节点已写正确 `@包名/组件名`。
- [ ] 不支持资源引用的节点没有强行写 `@资源引用`。
- [ ] 普通节点使用小驼峰，界面 / 组件使用大驼峰。
