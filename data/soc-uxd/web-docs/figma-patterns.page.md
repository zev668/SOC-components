# Figma 读写规范

这篇文档说明在 SOC 项目里，读取、评审或生成 Figma 交互稿时应该遵守什么工作方式。

它的核心不是“怎么调用工具”，而是保护团队资产：先读懂页面结构、历史组件和来源依据，再生成草稿；不要直接改正式文档、不要覆盖组件库主资产、不要把 AI 的推断当成已确认结论。

## 这篇规范解决什么问题

Figma 工作最怕两类错误：看得不够就动手，或者动手后没人知道依据是什么。

- 还没看页面结构，就直接生成新画板。
- 在正式交互稿或组件库里直接改节点，影响他人使用。
- 没查历史组件，重复造按钮、弹窗、导航和列表。
- 草稿里没有来源、假设和待确认项，评审时无法追溯。
- 移动端和 PC 行为混在一起，下游不知道按哪个平台实现。
- 生成后不截图、不读元数据，布局问题留到后面才暴露。

这篇规范的目标，是让 Figma 工作始终可追溯、可评审、可回退。

## 读 Figma 时先看什么

![Figma 读写规范路径](assets/soc-uxd/figma-patterns.svg)

读 Figma 不要只看截图，先建立上下文。

| 先看什么 | 为什么 |
|---|---|
| 页面结构 | 判断正式稿、草稿区、组件库、历史方案分别在哪里 |
| 选中节点和父级 | 确认当前任务范围，避免改错区域 |
| 命名和分组 | 识别 `bg / root / topBar / navBar / content` 等框架结构 |
| 已有组件 | 判断是否可以复用，而不是新建 |
| 历史标注 | 找到程序、GUI、拼接、动效注意事项 |
| 截图和元数据 | 互相验证画面是否和节点结构一致 |

如果只看一张图就开始改，很容易把参考图、说明图、正式节点和临时草稿混在一起。

## 写 Figma 时怎么保护源文档

默认规则：先建草稿，不动正式源。

| 操作 | 默认做法 |
|---|---|
| 生成新方案 | 放在 `AI草稿` 页面或清楚命名的草稿 Frame |
| 修改历史稿 | 只有用户明确要求同步正式页面时才改 |
| 引用组件库 | 先查已有组件和设计系统 |
| 写假设 | 直接放在草稿顶部或右侧说明里 |
| 写来源 | 保留源链接、源节点 ID、历史依据 |
| 生成后检查 | 用截图或元数据做一次验证 |

AI 生成内容必须让人一眼看出：这是草稿、依据是什么、哪些还没确认。

## 组件复用优先级

Figma 交互稿里，优先复用项目已存在的组件家族。

| 类型 | 优先查找 |
|---|---|
| 全屏 / 弹窗结构 | `ComModal`、`ComModal_FullScreenBelt-shape`、`ComTransparentSideShot`、`ComSideShot` |
| 选择侧栏 | `CommonSelectInventorySide`、`CommonSelectInventorySide2`、`UiExpandPopup` |
| 物品 / 文本 Tips | `UiItem Tips`、`UiHoverTip`、通用文本 tips |
| 导航 | `ComCommonNavBar`、`ComNavBar02-05`、`ComTopBar`、`ComTopBar_Title` |
| 按钮 | `ComStateBtn_*`、`ComPriceBtnNew`、`ComStateBtn_FreeResize` |
| 搜索 / 输入 | `ComSearch`、`ComInputBoxRoot`、`ComChangeName` |
| 数量 / 物品操作 | `UiItemSplit`、`UiCounterPopup`、步进器、背包格子 |
| 奖励 / 分享 / 结果 | `GlobalGetReward`、`CommonSharePop` |

如果需要新组件，交互说明里要写明：现有组件为什么不够用、新组件承担什么差异、是否影响 FGUI 和程序实现。

## 双端转译时怎么落到 Figma

移动端转 PC 时，不要把移动端界面等比放大。

在 Figma 里至少分清：

- 移动端和 PC 是否拆成独立区域。
- PC 是否重组 `topBar / navBar / content`。
- 移动端纵向 Tab 是否转成 PC 顶部横向导航。
- PC 的 `KeyPromptBar` 是否属于框架层，而不是业务内容区。
- 弹窗、Tips 是否属于全局临时层。
- PC 是否需要 hover、键盘提示、鼠标行为和 Esc 关闭。
- 该页面是纯适配，还是需要自定义 PC UI 布局。

双端差异要放在交互稿里，而不是让拼接或程序自己猜。

## AI 生成 Figma 草稿应该包含什么

当 AI 根据策划文档生成 Figma 草稿时，至少包含这些区域：

| 区域 | 内容 |
|---|---|
| 顶部摘要 | 来源、目标、范围、假设、历史依据 |
| 流程图 | 主路径、异常分支、返回路径 |
| 页面 / Frame 清单 | 移动端、PC、弹窗、状态页 |
| 状态板 | 默认、选中、hover、禁用、空态、加载、错误 |
| 规则标注 | 每个关键控件旁边的编号说明 |
| 职能注意 | 程序、GUI、拼接、动效、策划注意 |
| 待确认项 | AI 不能自行决定的问题 |

草稿越清楚，下游越容易评审；草稿越像正式稿，越要明确“AI 生成，待确认”。

## 交付前快速清单

- [ ] 已先读取页面结构、节点命名和历史组件。
- [ ] 没有直接改正式交互稿或组件库主资产。
- [ ] 新内容放在清楚命名的草稿区域。
- [ ] 草稿顶部写了来源、假设和历史依据。
- [ ] 优先复用了现有组件。
- [ ] 新组件需求写明原因。
- [ ] 移动端和 PC 行为分开表达。
- [ ] 已截图或读取元数据做生成后检查。
- [ ] 待确认项没有被写成确定规则。
