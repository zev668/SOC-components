# 全屏界面框架与移动转 PC 规范

这篇文档说明 SOC 项目里的全屏界面应该先怎么分框架，再怎么从移动端方案转译到 PC 方案。

它不讨论某个功能内容区的具体业务细节，也不处理弹窗、Tips、HUD、半屏界面和遮挡层级。它只回答一个更底层的问题：一个全屏页面的 `topBar`、`navBar`、`content` 应该如何构成，以及移动端的导航和内容在 PC 上应该如何重新组织。

## 这篇规范解决什么问题

很多双端界面的问题，来自一开始没有分清“框架”和“内容”：

- 移动端侧边导航被直接搬到 PC，导致内容区仍然很窄。
- PC 页面只是把移动端等比放大，没有利用横向空间。
- 返回按钮、标题、一级 Tab、二级 Tab 都被塞进顶栏，信息层级混乱。
- 某个页面到底算无导航、一级导航还是一二级导航，没有统一判断。
- 设计稿先画内容，后补导航，导致后续 FGUI 拼接和程序状态很难稳定。

这篇规范的目标，是让设计开始前先完成框架判断：先确认 `root` 下面有哪些模块，再进入具体内容排版。

## 什么时候先看框架

只要页面满足“全屏功能界面”这个前提，就先使用这篇规范。

| 场景 | 先判断什么 |
|---|---|
| 新建全屏功能页 | 是否需要 `navBar`，需要一级还是一二级 Tab |
| 移动端方案转 PC | 侧边导航是否进入 PC 顶部，内容区是否释放横向空间 |
| 旧界面改版 | 原有导航结构是否保留，哪些内容属于框架，哪些属于业务内容 |
| Figma 草稿评审 | 是否先画了正确的 `topBar / navBar / content` 骨架 |
| FGUI 拼接准备 | 节点结构是否能稳定映射到底层 View |

如果页面不是全屏界面，比如系统弹窗、Tips、HUD、半屏面板，就不要把本篇规则硬套上去。

## 全屏 View 的标准骨架

![全屏界面框架与移动转 PC 的骨架关系](assets/soc-uxd/fullscreen-framework-translation.svg)

全屏界面的底层 View 先分两层：

1. `bg.背景层`
2. `root.UI内容根节点`

`root` 下面再拆成三个框架模块：

- `topBar.顶栏`：标题、顶栏导航、货币列表、必要的顶栏状态。
- `navBar.一二级Tab`：可选模块，用来承载一级或二级功能分类。
- `content.内容`：功能主体区域，承载真正的业务信息和操作。

这里的关键是：`navBar` 不是默认存在的。它是否出现，要由当前页面的信息架构决定，而不是因为模板里有位置就保留。

## 三类全屏框架怎么判断

设计全屏页面时，先从导航复杂度判断框架类型。

| 类型 | root 下的结构 | 适用情况 |
|---|---|---|
| 无导航全屏 | `topBar + content` | 没有一级 / 二级功能 Tab，只需要标题和主体内容 |
| 一级导航全屏 | `topBar + navBar + content` | 有一级功能分类，但不需要二级 Tab |
| 一二级导航全屏 | `topBar + navBar + content`，其中 navBar 承载两级 Tab | 有一级分类，并且每个一级分类下还需要二级切换 |

判断顺序很简单：先问有没有一级分类；如果有，再问每个一级分类下是否还需要二级切换。不要先画按钮，再反推结构。

## 移动端转 PC 的核心原则

移动端转 PC，不是等比放大，也不是把侧边栏原封不动搬过去。

核心原则有五条：

- 移动端的功能结构要保留。
- PC 端的框架区域要重新适配。
- 原来被移动端侧边导航占用的空间，要尽量释放给 `content`。
- 一级导航通常转为 PC 顶部横向导航。
- `content` 必须根据 PC 画幅重新延展、填充或重排。

也就是说，移动端方案提供的是功能结构，不是 PC 尺寸下的最终布局。

## TopBar、NavBar 和 Content 怎么转译

**TopBar**
移动端顶栏里的标题、货币列表、返回按钮和横向 Tab，在 PC 上要重新判断归属。一级导航优先进入 `topBar` 横向展示；二级导航通常放在一级导航下方，作为跟随顶栏的导航区域。移动端返回按钮不要机械搬成 PC 顶栏固定内容。

**NavBar**
移动端左侧纵向导航转 PC 后，通常不再挤占内容区。一级导航进入顶栏，二级导航可以跟随顶栏横向排列。如果一级或二级导航超出显示区域，需要考虑横向滚动或收纳规则。

**Content**
`content` 是 PC 改造收益最大的区域。导航释放空间后，内容区应该按信息密度、操作频率和可读性重新适配，而不是保持移动端窄列结构。

## 不适合转入 TopBar 的例外

不是所有侧边结构都应该转成 PC 顶部导航。

如果某个侧边区域本质上是内容目录、筛选列表、长文本分类或复杂树结构，它更可能属于 `content` 的一部分，而不是标准 `navBar`。这类情况要在交互文档中说明原因，避免被误判为“漏做导航转译”。

判断方法很直接：如果它是功能分类入口，优先看作导航；如果它是内容阅读、筛选或编辑的一部分，优先看作内容。

## 交互输出时必须回答的问题

输出全屏界面交互稿前，至少回答这六个问题：

1. 这个页面是否属于全屏界面？
2. 它是无导航、一级导航，还是一二级导航？
3. `root` 下是否需要 `navBar`？
4. 移动端导航在 PC 上是否应该进入 `topBar`？
5. `content` 在 PC 画幅下如何动态适配？
6. 是否存在不适合转入 TopBar 的例外？

如果这些问题还没回答清楚，不要急着进入内容区细节。

- [ ] 已判断无导航、一级导航或一二级导航。
- [ ] 移动端导航转 PC 的位置和形态已说明。
