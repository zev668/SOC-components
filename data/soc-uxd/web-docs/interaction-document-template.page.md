# 交互文档模板与撰写规范

这篇文档说明 SOC 项目里的交互文档应该怎么组织、怎么标注、怎么交付。

它不是一份“填空格式”，也不是要求所有功能都写成同一种篇幅。它真正约束的是阅读路径：策划、交互、GUI、拼接、程序、动效都能在同一份 Figma 文档里快速找到自己需要的信息，并且能判断哪些内容已经确定，哪些内容还需要回到负责人确认。

## 这篇模板解决什么问题

交互文档最容易出问题的地方，往往不是画面本身，而是画面旁边的信息不够稳定：

- 页面截图有了，但不知道入口、返回、异常状态怎么走。
- 标了按钮，但右侧说明找不到对应编号。
- 写了“程序注意”，但没有说清楚触发条件、数据来源或失败处理。
- 下游只看到静态画面，不知道哪些是策划规则、哪些是交互推断。
- 旧模板被复制后，封面、日期、变更记录和人员信息没有更新。

这套模板的目标，是把交互文档从“界面截图合集”变成“协作交付物”。每一页不只说明长什么样，还要说明谁来处理、什么条件触发、系统如何反馈、后续如何落地。

## 什么时候使用这套模板

只要一个功能会进入正式设计、评审或开发流转，就应该使用这套模板。

| 场景 | 模板要保证什么 |
|---|---|
| 新功能交互稿 | 从封面、主画板、编号说明到交付检查都有稳定位置 |
| 旧功能改版 | 旧行为、保留逻辑、新增变化和待确认项能被分开阅读 |
| 多职能协作 | 程序、GUI、拼接、动效、策划注意事项能被明确标识 |
| Figma 草稿转正式稿 | 草稿判断、假设和引用依据不会混进最终规则 |
| AI 生成交互初稿 | AI 必须套用模板结构，并把不确定内容标为待确认 |

如果只是临时讨论一个概念，可以先画草图；一旦这份内容要被别人执行，就要回到模板。

## 交互文档应该长什么样

![交互文档模板的标准阅读路径](assets/soc-uxd/interaction-document-template.svg)

一份可交付的交互文档，至少要让读者按下面的顺序读下去：

1. 先看封面，确认文档名称、日期、变更记录和变更人。
2. 再看主画板顶部信息，确认功能名、协作角色和标识说明。
3. 看左侧界面或流程画板，理解页面、状态和跳转关系。
4. 看左侧编号，定位具体控件、区域或特殊状态。
5. 看右侧说明，理解每个编号的触发条件、系统响应和下游注意。
6. 最后用交付检查清单确认是否可以进入评审或开发。

这条阅读路径要尽量稳定。读者不应该靠猜测去判断“这段话对应哪个按钮”“这条线跳到哪里”“这个规则是谁负责”。

## 页面和编号怎么组织

模板的核心是“左图右文”。

左侧放界面、状态、流程画板或截图；右侧放与编号一一对应的文字说明。编号不是装饰，它是读者在画面和说明之间来回定位的锚点。

编号使用时遵守四条规则：

- 左侧有编号，右侧必须有对应说明。
- 右侧说明引用的编号，左侧必须能找到。
- 编号要靠近被说明的控件、区域或状态。
- 同一画板内的编号，按阅读顺序或操作流程排列。

右侧说明不要只写一句“点击后跳转”。一条可执行说明通常要包含：触发条件、系统响应、状态变化、跳转结果、异常边界，以及需要哪个职能继续处理。

## 职能注意事项怎么写

交互文档不是只给交互自己看的。只要说明影响到下游，就要加职能标识。

| 标识 | 适合标在哪里 |
|---|---|
| 程序注意 | 数据、接口、权限、倒计时、校验、红点、异常处理 |
| GUI 注意 | 图标、插画、背景、特殊视觉状态、资源补充 |
| 拼接注意 | FGUI 组件结构、控制器、适配、层级、资源引用 |
| 动效注意 | 入场、退场、按钮反馈、状态转场、循环效果 |
| 策划注意 | 数值、文案、奖励、解锁条件、配置字段 |
| 流程分支 | 跳转、返回、二次确认、异常流程、跨系统入口 |
| 文档更新 | 版本差异、待补充内容、后续变更记录 |

一个编号可以挂多个职能标识。比如一个领取按钮，可能同时需要程序确认奖励接口、GUI 补充图标状态、动效处理点击反馈、策划确认领取条件。

## 跳转连线和状态怎么表达

当界面之间存在跳转关系时，用连线表达路径，但不要只靠线解决说明。

连线负责告诉读者“从哪里到哪里”；右侧说明负责讲清楚“什么条件触发、跳到哪个状态、能不能返回、失败怎么处理”。

如果同一个入口有多个分支，要把分支条件写清楚。常见分支包括：未解锁、无权限、资源不足、网络失败、重复点击、已领取、已过期、跨系统返回。

## Figma 交付前怎么检查

交付前先做一轮自检：

- 封面是否已经替换模板占位信息。
- 主画板标题、日期、变更人和协作角色是否清楚。
- 左侧每个编号是否都有右侧说明。
- 右侧每条说明是否能定位到左侧编号。
- 关键流程是否有跳转连线和文字说明。
- 程序、GUI、拼接、动效、策划等下游事项是否被标识。
- 画板扩展是否只改变容器高度，没有缩放或移动原界面。
- 未确认内容是否明确标为待确认，而不是当成最终规则交付。

## 常见误区

**只贴截图，不写行为。**
截图只能说明静态状态，不能说明触发、反馈、异常和返回路径。

**编号和说明互相找不到。**
读者一旦需要猜编号，文档就失去了协作价值。

**把推断写成确定规则。**
AI 或交互可以补充建议，但奖励、解锁、配置、接口等决策必须标清来源或待确认。

**把所有注意事项写成一坨文字。**
不同职能要能快速扫到自己的工作，职能标识不是装饰，是分工入口。

## 交付前快速清单

- [ ] 文档封面已更新。
- [ ] 主画板信息已更新。
- [ ] 左侧画面、状态或流程足够覆盖本次需求。
- [ ] 所有关键控件、区域和状态都有编号。
- [ ] 编号与右侧说明一一对应。
- [ ] 跳转关系既有连线，也有文字说明。
- [ ] 下游注意事项已经按职能拆开。
- [ ] 待确认内容没有被写成确定结论。
- [ ] 文档可被策划、交互、GUI、拼接、程序、动效共同阅读。
