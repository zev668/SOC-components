# 界面层级与堆栈规范

这篇文档说明 SOC 项目里，一个界面应该放在哪个运行层级，以及交互文档里需要补哪些堆栈、关闭、阻挡和双端差异信息。

它不是全量界面表的替代品。全量表格、字段分布和具体界面清单仍保留在 AI 约束源里；网页稿负责提供一条更容易阅读的判断路径。

## 这篇规范解决什么问题

界面层级如果没有提前判断，后续会出现很难定位的问题：

- 弹窗和全屏界面混在一起，打开顺序和关闭顺序不稳定。
- HUD、局内覆盖层、全屏界面、Notice 都被当成普通弹层处理。
- 移动端点击关闭和 PC Esc 关闭没有分开说明。
- 输入阻挡、点击穿透、跟随全屏入栈隐藏没有标注。
- 新界面没有说明特殊状态，比如死亡、倒地、Loading、拍照、管理员工具。

层级规范的核心结论是：新建或评审任何界面时，先判断它属于哪个运行层级，再定义关闭方式、输入阻挡、是否跟随全屏入栈隐藏、移动 / PC 差异和特殊状态。

## 什么时候先判断层级

只要一个界面会被打开、关闭、覆盖、入栈或挡住操作，就要先判断层级。

| 场景 | 需要先确认什么 |
|---|---|
| 新建全屏界面 | 是否属于 `501 Full Screen`，是否跟随全屏栈隐藏 |
| 新建弹窗 / Tips | 是普通 `601 Popup`，还是 `901 PopupLoading` 或轻量 `701 Notice` |
| 局内玩法 UI | 是 `101 GamePlay`、`201 HUD`，还是 `301 GamePlayOverlay` |
| PC 适配 | `layer_PC` 是否需要覆盖移动端层级，Esc 是否关闭 |
| Loading / 拍照 / 管理工具 | 是否使用专用层，不要混入普通弹窗 |
| 水印 / 输入法 | 是否属于常驻全局层，最大数量和覆盖关系如何 |

判断层级不只是工程配置问题，它会影响玩家看到什么、能不能点击、按 Esc 关到哪里，以及新界面是否会把旧界面遮住。

## SOC 界面层级怎么读

![SOC 界面层级与堆栈关系](assets/soc-uxd/interface-layering-spec.svg)

层级阅读时先记三条原则：

1. 层级排序数字越大，默认越靠上显示。
2. 后打开的界面会覆盖先打开的界面。
3. 关闭界面时，从最上层界面开始依次关闭。

常用层级可以这样理解：

| 层级 | 适合放什么 | 设计判断 |
|---|---|---|
| `101 GamePlay` | 准星、血条、世界标记、伤害跳字 | 与 3D 世界强绑定 |
| `201 HUD` | 玩家持续操作的 HUD 信息与按钮 | 具体元素要看 HUD 专表 |
| `301 GamePlayOverlay` | 局内半屏、玩法信息、覆盖在 HUD 上的信息层 | 不完全遮住场景 |
| `501 Full Screen` | 背包、商城、地图、设置等主系统界面 | 完全盖住场景 |
| `801 Loading` | 进入大厅、进入局内等加载界面 | 通常阻挡输入，不随意 Esc |
| `901 PopupLoading` | Loading 期间需要特殊共存的弹窗 | 要说明和 Loading 的遮挡关系 |
| `601 Popup` | 普通弹窗、确认框、二级 / 三级弹层 | 默认弹层承载 |
| `701 Notice` | toast、击杀播报、轻量系统提示 | 不抢占普通弹窗堆栈 |
| `902 Input` | 输入法相关全局层 | 常驻，最大数量为 1 |
| `910 WaterMark` | 测试水印等全局最高展示 | 全局覆盖 |

Pandora、Photo、Admin 属于专用层。普通界面不要因为“看起来差不多”就放进去。

## 设计时怎么选层级

设计师不需要先背所有层级编号，先按问题判断：

| 判断问题 | 倾向层级 |
|---|---|
| 是否和 3D 世界强绑定，比如准星、血条、世界标记？ | `101 GamePlay` |
| 是否是玩家持续操作的 HUD？ | `201 HUD` + HUD 专表 |
| 是否是局内覆盖信息，但没有完全盖住场景？ | `301 GamePlayOverlay` |
| 是否完全盖住场景，成为主系统界面？ | `501 Full Screen` |
| 是否是进入场景或进入大厅的加载？ | `801 Loading` |
| 是否是普通确认框、二级弹窗、Tips？ | `601 Popup` |
| 是否只是轻量通知、toast、播报？ | `701 Notice` |
| 是否为输入法、拍照、管理员工具、水印？ | 使用专用层 |

如果一个界面同时像两个层级，不要凭视觉猜。交互稿里要写明判断依据，并标为待确认或程序注意。

## 字段和双端差异怎么标

交互文档里至少要关注这些字段：

| 字段 | 设计师要说明什么 |
|---|---|
| `layer` | 默认 / 移动端所属层级 |
| `layer_PC` | PC 是否需要不同层级 |
| `HideWithStack` | 是否跟随全屏界面入栈隐藏 |
| `EscClose` | PC 按 Esc 是关闭、阻挡还是无效 |
| `IsBlockInput` | 是否屏蔽后层操作或点击穿透 |
| `isHalfScreen` | 半屏是否仍按玩法覆盖层或弹层处理 |
| `fullScreenVisually` | 视觉全屏但工程层级是否不同 |
| `showCursorType` | PC 是否显示鼠标与显示类型 |
| `showInDeath / showInWounded` | 死亡 / 倒地状态是否继续显示 |
| `needCloseGyro` | 移动端打开后是否关闭陀螺仪 |

移动端和 PC 不要默认一致。PC 常见问题是 Esc 行为、鼠标显示、层级覆盖、宽屏黑边和输入法层；这些都应该在交互文档里写清楚。

## HUD 为什么要单独看

HUD 不应只看 `UI界面表`。`UI界面表` 里只有 `UiHud` 位于 `201 HUD`，真正的 HUD 元素由 HUD 专表控制。

涉及 HUD 时，交互文档需要额外说明：

- HUD 元素名称。
- 移动端和 PC 是否不同。
- 是否阻挡点击。
- 内部排序、分组和内部层级。
- 是否可编辑、可移动、可缩放、可调透明。
- 不同玩法或状态下显示 / 隐藏规则。
- PC 键位引导和操作槽位。

把 HUD 当普通弹窗处理，会导致层级、输入和编辑状态都出错。

## 当前需要特别确认的差异

源规范里已经记录了几处需要后续确认的差异，网页也保留提醒：

- `Admin` 管理员工具层在规范图和 Excel 排序里不一致，需要确认最终口径。
- `Input` 输入法层存在于 Excel，图中未展示；它与 Notice 当前排序相同，需要确认 tie-breaker。
- `PandoraType0/1/2` 是专用承载层，普通界面不要放。
- `PopupLoading` 与 `Loading` 的共存关系需要按具体场景写清。

这些不是设计师单独拍板的内容，需要回到程序、配置或负责人确认。
