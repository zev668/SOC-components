# 多语言与本地化设计规范

这篇规范用于回答一个容易被低估的问题：多语言不是“翻译回来之后再修 UI”，而是在交互和 GUI 阶段就要提前设计好文本、布局、术语、资源和验证方式。

SOC 的界面会同时面对手游、PC、局内、局外、商业化、活动、聊天、队伍、HUD 等不同场景。如果只按中文最短文本画界面，等英文、泰语、繁中或后续更多语言接入后，最先暴露的往往不是翻译问题，而是按钮爆框、标题截断、行高不够、变量丢失、术语混乱和地区资源无法替换。

## 这篇规范解决什么问题

多语言设计的目标不是把所有语种都提前翻译一遍，而是让界面结构先具备“可被翻译、可被拉伸、可被验证、可被替换”的能力。

在 SOC 项目里，多语言相关返工通常来自下面几类问题：

- 中文按钮很短，英文或泰语接入后按钮、页签、标题直接爆框。
- 文本底图是静态切图，无法跟随文本长度做三宫或九宫拉伸。
- 行高、上下留白不足，泰语、阿拉伯语等语言出现顶部或底部截断。
- 占位符、热键、颜色标签、资源标签被翻译或伪本地化误改，导致程序解析失败。
- 同一个术语在局内、局外、商业化、系统提示里被翻成不同含义。
- 节日、商业化、剧情、语音、字幕等地区化资源没有预留替换结构。

这篇规范的作用，是让设计师在画稿阶段就把这些风险显式处理掉。能靠布局解决的不要留给翻译，能靠标注说明的不要留给拼接猜，能靠伪本地化提前暴露的不要等上线前才发现。

## 什么时候必须考虑多语言

只要界面包含可变文本、地区资源、术语或用户生成内容，就应该把多语言当成设计输入，而不是最后一轮检查项。

| 场景 | 必须考虑什么 |
|---|---|
| 按钮、页签、标题、标签 | 短文本通常比长段落更容易爆框，至少按中文 2 倍空间预留 |
| 说明文本、规则文本、剧情文本 | 需要确认换行、行高、滑动区域和阅读节奏 |
| 弹窗、toast、tips、系统提示 | 要保证核心信息完整可读，不能靠硬截断兜底 |
| 列表、排行榜、昵称、物品名 | 需要决定省略、tooltip、详情页或滚动展示方式 |
| 商业化、活动、节日资源 | 要区分通用资源和地区替换资源，避免文字做死到图片里 |
| 热键、变量、颜色、资源标签 | 必须保留占位符和富文本结构，不参与翻译改写 |
| PC / 移动双端 | PC 还要额外检查 hover、宽度拉伸、快捷键提示和信息密度 |

一句话判断：如果文本变长、换语种或换地区资源会影响界面理解，就必须在交互稿里标注多语言处理方式。

## 多语言适配的设计路径

![多语言适配的设计路径](assets/soc-uxd/multilingual-flow.svg)

这条路径的关键点是：不要等翻译完成后才开始验证。设计阶段先用规则预留空间，交付阶段明确适配方式，验证阶段用伪本地化提前压测。

| 阶段 | 设计师要做什么 | 输出结果 |
|---|---|---|
| 源中文清理 | 把中文文案写成可翻译、少歧义、少文化梗的表达 | 可被稳定翻译的源文案 |
| 结构预留 | 为按钮、标题、段落、列表、弹窗预留宽度和高度 | 不依赖中文最短长度的布局 |
| 适配标注 | 标明每类文本使用拉伸、换行、收缩、省略、滚动或组合方案 | 下游可执行的拼接规则 |
| 伪本地化压测 | 在 Figma 或 Unity 中用膨胀文本提前检查截断、重叠、行高 | 可回归的问题清单 |
| 术语与地区资源交付 | 标注术语来源、冲突项、地区资源后缀和替换规则 | 翻译、GUI、拼接、程序可对齐的交付包 |

## 文本空间应该怎么预留

不要用中文最短文本决定控件尺寸。中文在很多 UI 里是“最省空间”的语言，按中文贴边设计会把后续语种全部推入风险区。

| 目标语言 | 长度趋势 | 建议预留比例 | 设计含义 |
|---|---:|---:|---|
| 英语 | 略长 | 1.2x 至 1.5x | 常规按钮、标题、说明至少留出扩展空间 |
| 日语 | 较短但占位接近中文 | 0.9x 至 1.0x | 不要因为日语短就压缩通用组件 |
| 泰语 | 略长且垂直结构复杂 | 1.5x 至 2.2x | 同时检查宽度、行高、顶部和底部截断 |
| 德语 | 长词多，结构复杂 | 2.0x 至 3.0x | 若后续接入，需要按极端长度设计关键容器 |
| 俄语 | 长词多，结构复杂 | 2.0x 至 3.0x | 若后续接入，需要按极端长度设计关键容器 |

短文本要更保守。按钮、页签、标题、标签这类短文本，建议按中文长度的 2 倍预留；长段说明至少按 1.5 倍预留。未来若接入长词语言，关键容器需要考虑 3 倍极端情况。

跟随文本变化的视觉元素也要一起适配：按钮底图、标签底图、文本两侧 icon、装饰边框、价格区域和奖励信息，都不能只做静态贴图。文本底图应优先支持三宫或九宫拉伸；两侧 icon 要跟随文本宽度联动，而不是固定死在中文位置上。

多行文本要同时检查横向和纵向。横向太窄会把一个完整句子拆成碎片，纵向太紧会导致泰语、换行符或富文本显示拥挤。规则说明、剧情对白、长 tips 等内容，优先使用高度自适应或外层视窗裁切加滑动框，不要直接硬截断。

## 适配方式怎么选

交互稿不能只写“注意多语言”。它必须告诉下游：这个文本超出时，到底是拉伸、换行、收缩、省略、滚动，还是组合处理。

| 适配方式 | 适合场景 | 使用规则 |
|---|---|---|
| 长度自适应 | 按钮、标签、短提示、可伸展底图 | 文本框、底图和两侧 icon 要联动扩展 |
| 高度自适应 | 说明文本、列表描述、段落内容 | 容器高度随文本增加，并让相邻内容避让 |
| 收缩 | 限定高宽、短文本不可换行时 | 可作为默认兜底，但必须设置最小字号下限 |
| 省略号 | 列表、排行榜、昵称、非关键长文本 | 只用于用户可通过 tooltip、详情页或上下文补全的信息 |
| 滚动 / 跑马灯 | 少数无法通过宽高解决的展示文本 | 只用于特殊展示，不作为普通 UI 默认方案 |
| 上下滑动 | 大段说明、规则、剧情文本 | 文本高度自适应，外层视窗裁切并提供滑动 |
| 截断 | 不推荐 | 除装饰性内容外，不应作为交付方案 |

如果一个文本会影响操作决策，例如按钮、价格、奖励、权限、倒计时、确认提示，不要默认省略。优先保证完整读到，再考虑排版美观。

## 占位符、富文本和术语怎么处理

多语言最容易出错的不是普通句子，而是夹在句子里的变量和标记。交互稿需要提前标注这些内容的来源和保留规则。

| 类型 | 示例 | 处理规则 |
|---|---|---|
| 占位符 | `{0}`、`{hotkey=Skip}` | 全量保留，不参与翻译改写，不计入伪本地化目标长度 |
| 富文本标签 | `[color=#FFC02A]`、`[/color]` | 保留结构，前后文本独立计算长度后按原位置填回 |
| 资源标签 | `<img src='ui://...'/>`、`<br>` | 保留标签，不让翻译或伪本地化破坏程序解析 |
| 换行符 | `\n`、`\r\n` | 明确是强制换行还是文本自然换行，不要混用 |
| 热键 | `Esc`、`Enter`、`F` | 标注键位来源，避免被当普通文案翻译 |

术语必须看场景，不是机械套一个翻译。比如 `market.open` 在局内语境可能是 `Shop`，局外商业化商店可能是 `Store`；`animal_equipment` 指马鞍等动物装备时可以用 `Gear`，玩家装备语境才更接近 `Equipment`。

如果客户术语、Rust 原术语和 SOC 实际语境冲突，交互稿里要写清楚：使用场景、冲突来源、建议用词、待确认人。不要把冲突藏在文案里等翻译后再修。

## 伪本地化怎么验证

伪本地化用于在正式翻译返回前，提前发现界面里的爆框、重叠、截断、行高、富文本、占位符和极端语言问题。

| 验证位置 | 适合发现什么问题 |
|---|---|
| Figma 阶段 | 文本扩展后是否爆框，底图是否能拉伸，图标是否跟随文本移动 |
| Unity / 游戏内 GM 模式 | 实际字体、实际拼接、真实状态下是否截断或重叠 |
| 泰语压力测试 | 顶部声调、上元音、基底、下元音是否造成上下截断 |
| 占位符与富文本测试 | 变量、颜色、资源标签、换行符是否被保留 |

游戏内伪本地化入口为 GM 指令，格式是 `语言代码;伪本地化模式ID`。例如 `zhtw;0` 表示切换繁体中文并使用模式 0，`arsa;1` 表示切换阿拉伯语并使用模式 1。

伪本地化文本会按中文原文长度乘以倍率生成，并用首尾英文感叹号 `!` 标识，便于区分拼接文本。若语言配置需要插入空格，应按配置频率插入，用于测试安全换行、多行排版和行高。

泰语不能随机捞取单个字符。它需要按“基底 + 元音 + 声调”的视觉单位成组生成，才能真实测试 UI 顶部和底部是否会被截断。

## 本土化资源怎么交付

文字不是唯一需要本地化的内容。节日活动、商业化图素、剧情图文、字幕、语音、地区禁忌图标，都可能需要替换。

| 类别 | GUI 关注点 | 交互关注点 |
|---|---|---|
| 节日活动 | Banner、节日符号、活动图素支持地区替换 | 活动入口结构不要和单一地区资源绑定 |
| 商业图素 | 礼包卡、标签、价格区、促销图层可变 | 组件预设支持不同版本结构 |
| 剧情图文 / 语音 | 对话、字幕、语音资源可替换 | 文本与语音分轨，避免强绑定 |
| 通用标记 | “新”、折扣、限时、热卖等标记可替换 | 判断是否可用统一英文资源兜底 |

相同使用场景的资源建议使用地区后缀，例如 `Chs`、`Eng`。如果功能不做扩展，且语义简单、无文化风险，可以统一使用 `Eng` 后缀资源作为海外兜底；但节日、宗教、文化符号、商业促销等内容不能默认通用。

## 交付前快速清单

- [ ] 按钮、标题、页签、标签等短文本是否按中文 2 倍空间预留。
- [ ] 长说明文本是否至少按中文 1.5 倍空间预留。
- [ ] 泰语、德语、俄语等极端语言是否有高度、行高或滑动兜底。
- [ ] 文本底图是否支持三宫或九宫拉伸。
- [ ] 文本两侧 icon、价格区、奖励区是否跟随文本宽度联动。
- [ ] 每类风险文本是否标明适配方式，而不是只写“多语言适配”。
- [ ] 是否避免把硬截断作为默认方案。
- [ ] 占位符、热键、富文本、资源标签是否标注保留规则。
- [ ] 术语冲突是否写清楚来源、场景、建议用词和待确认人。
- [ ] 地区替换资源是否有后缀、加载优先级或 fallback 规则。
- [ ] Figma 阶段是否用伪本地化或文本替代做过溢出检查。
- [ ] Unity 阶段是否通过伪本地化 GM 模式做过界面回归。
