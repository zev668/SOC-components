# 信息提示系统

这篇文档说明 SOC 项目里，遇到系统反馈、错误反馈、奖励反馈、目标提示、新手引导和持续提醒时，应该怎么选提示类型、怎么标配置来源、怎么写队列优先级和双端差异。

它不是提示样式图库。真正要先判断的是：这条信息有多重要、会不会打断玩家、持续多久、是否需要跳转查看、是否进入队列、是否会覆盖其他提示，以及移动端和 PC 的行为是否不同。

## 这篇规范解决什么问题

提示系统最容易出问题的地方，是把所有反馈都当成“弹一下文字”：

- 操作失败只写 Toast，没有说明错误码和触发条件。
- 重要获得提示和普通获得提示混用，玩家不知道是否可点击查看。
- 持续提示、短触发提示、侧弹窗提示互相覆盖，但交互稿没有写队列规则。
- PC 需要快捷键跳转，移动端需要点击定位，但文档里没有区分。
- 新手引导中某些提示应隐藏，却没有标 `isGuideHide`。
- 新增提示没有查重 `constName`，后续程序和配置容易冲突。

这篇规范的目标，是让交互稿把“提示为什么出现、出现多久、如何消失、是否覆盖、如何配置、谁来实现”写清楚。

## 什么时候使用提示系统

只要功能里出现“告诉玩家某件事”的反馈，都应该先查提示系统。

| 场景 | 先判断什么 |
|---|---|
| 操作成功 / 失败 | 是否只是轻量反馈，是否需要错误码 |
| 条件不足 | 是通用文本提示，还是界面内禁用说明 |
| 战斗中反馈 | 是否需要低打扰，不抢占操作 |
| 任务目标变化 | 是短触发目标提示，还是持续目标提示 |
| 获得物品 / 科技 | 是普通样式，还是重要样式，是否可查看 |
| 新手教学 | 是否使用 NPC 文本或键位引导，是否引导中隐藏 |
| 关键节点 / 成就 | 是否高优先级覆盖其他提示 |

如果反馈会影响玩家下一步行为，就不能只写一句文案。要把触发条件、展示方式、消失条件和下游配置都写出来。

## 提示类型怎么选

![信息提示系统的选型路径](assets/soc-uxd/prompt-system.svg)

常用提示类型可以按强度和场景理解：

| 提示类型 | 适合场景 | 设计判断 |
|---|---|---|
| 通用文本提示 | 成功、失败、条件不足、轻量操作反馈 | 短时、低打扰、不阻断 |
| 战斗弱提示 | 局内战斗中的弱反馈 | 不抢占战斗操作 |
| 常驻准星提示 | 准星、瞄准、局内持续状态 | 更接近玩法层，不按普通 Toast 处理 |
| 短触发提示 | 阶段目标、任务目标、短时间强调 | 展示明确目标，时间较短 |
| 获得物品提示 | 道具获得、失去、奖励获得 | 多数由服务器下发后触发 |
| 左侧信息弹窗提示 | 首次获得、重要物品、科技提示、倒计时信息 | 权重更高，可承载查看 / 跳转 |
| NPC 文本提示 | 新手引导中的对话或提示 | 需要确认是否绑定引导系统 |
| 持续性提示 | 持续目标、倒计时、局内长期提醒 | 必须写队列、覆盖、消失条件 |
| 关键节点提示 | 情报柜、科技、评级等关键节点 | 高优先级，会压过其他提示 |
| 奖励提示 | 奖励展示、结算前后反馈 | 要写奖励来源和展示时机 |
| 新手键位引导 | PC / 移动操作教学 | 要写平台差异和键位信息 |

选型原则是：能用轻提示解决，就不要上强提示；需要玩家处理或跳转查看的，再使用更高权重的提示。

## 提示强度和队列怎么判断

提示不是单独出现的，它会和其他提示共享屏幕、队列和注意力。

| 判断问题 | 应该写清什么 |
|---|---|
| 是否会覆盖其他提示？ | 覆盖对象、恢复时机、是否强制打断 |
| 是否进入队列？ | 队列类型、同类覆盖、最大显示数量 |
| 是否有 CD？ | 相同提示多久内不重复出现 |
| 是否自动消失？ | 展示时长、倒计时、手动关闭方式 |
| 是否可点击查看？ | 点击行为、跳转目标、返回路径 |
| 是否引导中隐藏？ | `isGuideHide` 与设置项条件 |

持续性提示、重要节点提示、重要成就提示尤其要写清楚。它们可能暂时隐藏普通提示或持续提示，消失后再恢复其他队列。

## 配置来源和字段怎么标

交互稿里不要只写“弹提示”，要写配置来源。

| 配置来源 | 主要覆盖 | 交互稿要标什么 |
|---|---|---|
| `05_通用提示表` | 通用文本提示、战斗弱提示、物品获得、准星提示、NPC 文本 | 提示 ID、文案、时间、CD、图标、颜色、音效、引导隐藏 |
| 大厅杂项 / 错误码表 | 错误码提示 | 错误码 ID、模块、style、错误文案 |
| `189_持续性提示表` | 持续性提示、新手键位引导 | 类型、时间、标题、主文案、副文案、倒计时、图标 |
| `192_短触发目标提示表` | 短触发目标提示 | 提示 ID、样式 type、标题、时间、背景、音效 |
| `193_侧弹窗提示表` | 侧弹窗、倒计时侧弹窗 | ID、名称、样式、标题、文案、时间、CD、倒计时 |
| `41_道具_总表` | 物品首次获得提示 | 是否显示引导通知 ID、普通 / 重要样式 |
| `97_系统_研发表` | 科技首次获得提示 | 普通 / 重要样式、查看 / 跳转行为 |

新增提示时，还要说明是否需要程序新增常量、是否需要在 `CommonTipConst` 查重、是否需要新增 FGUI 资源或配置字段。

## 移动端和 PC 差异怎么写

提示系统是双端差异高发区。交互稿至少要写：

- 移动端是否点击查看，点击后跳转到哪里。
- PC 是否有快捷键跳转，快捷键提示放在哪里。
- PC 是否可 Esc 关闭侧弹或确认提示。
- 鼠标、键盘、手柄输入是否影响提示显示。
- 同一提示在移动端和 PC 的位置、宽度、时长是否一致。

物品首次获得、科技首次获得和新手键位引导尤其要分开写。移动端常常是触控和定位，PC 常常是快捷键、键位提示和鼠标行为。

## 程序与 FGUI 落地注意

交互稿需要给下游留下可执行信息：

- 通用即时提示通常通过 `Mc.MsgTips` 对应方法调用。
- 错误码提示走错误码配置，`style` 为 `MsgTip` 时调用通用提示。
- 获得道具多数由服务器下发后触发，不一定需要手动调用。
- 音效需要写 `audioEvent`。
- 背景图片能力有限的提示类型，不要在设计稿里强行要求换背景图。
- 涉及侧弹窗、持续提示、重要节点提示时，要额外写清覆盖和恢复逻辑。

如果设计稿需要新增样式，不要只画效果图；要说明现有提示类型为什么不够用，以及新增影响哪些配置、资源和程序接口。
