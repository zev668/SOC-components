# 通用组件复用规范

这篇文档说明 SOC 项目里，做新界面或改旧界面时，什么时候应该先查通用组件，怎么判断组件能不能直接复用，以及交互稿里需要把哪些使用边界写清楚。

它不是完整组件百科。完整清单仍以源 Excel、Figma 组件库和 AI 约束源为准；网页稿负责把查阅路径讲清楚，让设计、GUI、拼接和程序能用同一套判断方式沟通。

## 这篇规范解决什么问题

通用组件的价值不是“少画几个控件”，而是让同类交互在不同系统里保持一致。

- 同样是确认按钮，不同界面不要各自定义尺寸、颜色和状态。
- 同样是物品选择，不要每个系统重新做一套选择弹窗。
- 同样是 Tips、空态、搜索、步进器和货币栏，要复用已有交互边界。
- 移动端转 PC 时，hover、键盘、滚轮、右键菜单和底部键位不要漏掉。
- 组件还没验收时，交互稿必须标注“待验收 / 待确认”，不能当作已完成能力写给下游。

这篇规范的目标，是让“复用”变成一个可检查的步骤，而不是评审时临时想起来的问题。

## 什么时候先查通用组件

![通用组件复用路径](assets/soc-uxd/soc-common-components-catalog.svg)

只要需求里出现下面任意一种情况，就先查通用组件或通用界面。

| 需求信号 | 优先查什么 |
|---|---|
| 需要完成一段完整流程 | 通用界面，例如物品选择、数量拆分、分享、奖励获得、改名、搜索用户 |
| 只是局部控件 | 通用组件，例如按钮、导航、标题、输入框、搜索框、空态、货币、物品卡片 |
| 是弹窗或侧弹 | 弹窗底板、半屏侧弹、透明 HUD 侧弹、二级操作列表 |
| 是道具、资源、奖励 | 物品卡片、物品 Tips、奖励弹窗、资源图标、数量步进器 |
| 是 PC 适配 | PC 标题栏、快捷键、滚动条、hover Tips、菜单、底部键位 |
| 是提示或引导 | 信息提示系统、气泡、倒计时提示、新手引导框、顶部可跳转提示 |

如果已有组件能承载需求，优先复用；如果不能，交互稿里要说明“不复用的原因”和“新增组件差异”。

## 组件复用怎么判断

判断组件能不能复用，按三个问题走。

| 判断问题 | 通过标准 |
|---|---|
| 场景是否一致 | 组件原本解决的问题和当前需求一致，不只是长得像 |
| 信息量是否一致 | 标题、图标、数量、品质、消耗、状态、按钮数量能放下 |
| 行为是否一致 | 点击、hover、长按、拖拽、快捷键、关闭方式、错误反馈能覆盖 |

常见误区是“视觉像就直接套”。SOC 里更应该看行为：按钮有没有禁用态，物品卡有没有选中态，Tips 有没有边缘避让，弹窗关闭是否等同取消，PC 是否支持键盘和鼠标。

如果只有视觉一致，但行为边界不一致，可以复用样式方向，不能默认复用完整组件。

## 组件成熟度怎么看

源表里的状态决定了交互稿的写法。

| 成熟度 | 交互稿写法 |
|---|---|
| 已验收 | 可以优先直接复用，写清 FGUI 名称和调用状态 |
| 待验收 | 可以作为依据，但必须标注待验收，并提醒拼接 / 程序核验 |
| 已封装待调整 | 可以参考结构，必须补齐缺失状态、适配规则或调整点 |
| 文档完成 | 只能引用规则，不能默认 FGUI 或程序已完成 |
| 计划增加 | 不能当作现成组件调用，只能写成新增组件需求 |
| 状态为空 | 使用前回查 Figma / FGUI 或找负责人确认 |

这张表的重点是防止“计划中组件”被写成“已可落地组件”。交互稿越接近执行，成熟度标注越不能省。

## 常用组件怎么选

下面是设计时最常查的组件族。网页只放选择依据，具体名称、数量和 FGUI 字段看底部源文档。

| 组件族 | 选择依据 |
|---|---|
| 通用界面 | 当需求需要承载完整流程时优先看，例如选择、拆分、分享、奖励、改名、搜索 |
| 按钮 | 先按操作层级选：主行动、一二三级、图标按钮、价格按钮、危险操作按钮 |
| 导航 | 先判断是全屏导航、内容区分页、横向 Tab、纵向导航还是阶段导航 |
| 弹窗底板 | 先判断打断强度和承载内容：大弹窗、普通弹窗、半屏侧弹、二级操作列表 |
| 标题与背景 | 先对齐全屏框架和弹窗结构，不把背景底板当内容卡片使用 |
| 基础控件 | 输入、搜索、下拉、单选、复选、空态、红点、货币、头像等优先查通用件 |
| 物品卡片 | 先判断信息密度：只展示图标、展示详情、队列中、品质资源或可选卡片 |
| PC 定制组件 | 只要出现 hover、快捷键、滚轮、右键、底部键位，就必须查 PC 组件 |

组件选择完成后，交互稿至少补齐：使用场景、触发方式、状态、平台差异、FGUI 名称和职能注意。

## 交付前快速清单

- [ ] 新界面已经先查通用界面，再查局部组件。
- [ ] 组件不是只按视觉相似复用，而是确认了行为和信息量。
- [ ] FGUI 名称、成熟度和待确认项写清楚。
- [ ] 禁用、选中、加载、空态、错误、数量不足等状态没有遗漏。
- [ ] PC 的 hover、快捷键、滚轮、菜单、底部键位已经单独检查。
- [ ] 新增组件写明现有组件为什么不够用。
- [ ] 需要 GUI、拼接、程序确认的问题没有被写成确定结论。
