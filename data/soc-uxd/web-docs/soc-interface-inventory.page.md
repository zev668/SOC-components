# 全量界面资产库

这篇资产库把 SOC 已整理的界面按系统大类和系统名称归档，并为每个系统保留手游图示、PC 图示和界面分析三类位置。

它的作用不是替代 Figma，也不是展示最终截图，而是让设计师快速知道：某个系统有没有历史界面、属于什么框架、常见状态是什么、跳转逻辑是否已经有初步判断。

## 这篇资产库解决什么问题

全量界面很多，如果只靠文件夹和截图记忆，很容易漏掉历史方案。

- 做新需求前，不知道同系统有没有旧页面。
- 做双端适配时，不知道 PC 是否纯适配、部分定制还是完全定制。
- 做组件复用时，不知道该页面更像 HUD、全屏、半屏、弹层还是列表详情。
- 写交互稿时，状态和跳转逻辑需要一个初始检查入口。

全量界面资产库就是这个入口：先帮你定位，再指向更具体的源材料。

## 界面资产按什么口径整理

![全量界面资产库](assets/soc-uxd/soc-interface-inventory.svg)

| 字段 | 作用 |
|---|---|
| 系统大类 | 先按局内、局外、全局通用等大方向归档 |
| 系统 | 具体系统或功能域，例如背包、制造、地图、组队、商城 |
| 手游界面图示 | 后续补移动端截图、Figma 节点或嵌入图 |
| PC 界面图示 | 后续补 PC 截图、Figma 节点或嵌入图 |
| 界面分析 | 固定记录框架、元素状态和跳转逻辑 |

源表里还保留了 PC 备注，用来提示纯适配、适配优化、部分功能定制、完全定制等差异。

## 查历史界面时看什么

查这张表时，重点不是“有没有图”，而是先建立页面判断。

| 先看什么 | 设计判断 |
|---|---|
| 界面框架 | 是 HUD、全屏系统、半屏容器、全局弹层、列表详情还是流程页 |
| 元素状态 | 是否涉及选中、切换、hover、快捷键、弹窗、Tips、加载或详情 |
| 跳转逻辑 | 是否已有入口、列表、详情、确认、结果和返回路径 |
| PC 备注 | PC 是纯适配、框架调整、表现定制还是功能定制 |
| 待补内容 | 哪些图示、状态矩阵或跳转仍需回到 Figma / 策划确认 |

如果一行只给了待确认，不要把它写成确定规则；它只是提醒你下一步该查哪里。

## 什么时候回到源表

网页稿只适合阅读和导航，下面这些情况必须回到源文件或 Figma。

| 情况 | 回源目的 |
|---|---|
| 需要截图或节点链接 | 网页稿不承载完整图示 |
| 要确认 PC 定制等级 | 以源表和最新实现状态为准 |
| 要做具体交互稿 | 需要结合策划输入、Figma 读写规范和组件规范 |
| 表内提示待确认 | 需要查 Figma、源 Excel 或找负责人确认 |
| 发现新页面 | 应回写到资产库，而不是只放在个人草稿里 |

资产库越常更新，AI 和设计师越不容易重复发明同一套界面。
